# 送检批次工作台 — 设计文档

## 1. 问题诊断

### 1.1 用户真实工作流

```
检查到期 → 整理待检清单 → 找备表 → 填检定申请表 → 送检
                                                          ↓
张贴管理一览表 ← 更新台账 ← 逐个核对 ← 收到检定证书 ← 检定完成
```

### 1.2 核心痛点

| 痛点 | 现状 | 根因 |
|------|------|------|
| 核对证书效率极低 | 收到 30 份证书 PDF，需逐个打开、看编号、在台账中搜索、对比安装位置、确认无误后手动编辑更新 | 证书和台账之间没有**并排对比视图** |
| 送检前后信息断裂 | 检定申请表导出了，证书回来后不知道哪些是同一批、哪些漏了证书 | 没有**批次**概念把送检→返回串起来 |
| 错误难发现 | 证书编号和台账编号差一个数字（录入错误），批量上传标记为 unmatched，用户需要离开当前页去编辑 | 修正操作不在**匹配上下文中** |
| 管理一览表更新滞后 | 更新完台账后还要另外导出管理一览表 | 一览表导出和证书更新是两个独立操作 |

### 1.3 参考文件分析

| 文件 | 内容 |
|------|------|
| `台账总表.xlsx` | 13 Sheet（压变 145 条、温变 54 条、压力表 220 条、温度表 44 条、流量计等），每个 Sheet 列结构不同 |
| `管理一览表.xlsx` | 14 Sheet，现场张贴用，字段：序号/器具名称/出厂编号/安装位置/状态/检测日期/检验周期/有效日期/管理标识/备注 |
| `检定申请表.xlsx` | 2 Sheet，送检清单，含器具名称/规格型号/准确度/测量范围/出厂编号/厂家/安装地点 |

## 2. 核心概念：送检批次

不引入复杂的批次管理系统。利用**台账导出→证书回传→并排核对→一键更新**的自然流程，在现有功能之上增加一个"工作台"视图。

### 2.1 流程改造

```
 预警页                     工作台                       台账列表
 ┌──────────┐           ┌──────────────────┐           ┌──────────┐
 │ 筛选到期  │──选择──→  │ 1. 生成检定申请表  │           │          │
 │ 器具列表  │           │ 2. 上传检定证书    │──确认──→  │ 更新完成  │
 │          │           │ 3. 并排核对匹配    │  更新    │          │
 │          │           │ 4. 导出管理一览表  │           │          │
 └──────────┘           └──────────────────┘           └──────────┘
```

工作台是一个**临时工作区**（不持久化批次数据），用户从预警页选好器具后进入工作台，在工作台内完成全部操作。

### 2.2 为什么不持久化批次

- 用户的实际操作模式是"一次送检、一次性处理完"
- 不存在"创建批次→过几天再回来处理"的场景（证书回来后当天就会处理完）
- 持久化会引入批次状态管理、过期清理等复杂度
- 工作台的数据来源是用户**当前选中的器具集合**，通过 URL 参数或 sessionStorage 传递

## 3. 工作台页面设计

### 3.1 入口

从**预警页**进入：勾选到期器具 → 点击"送检工作台"按钮 → 跳转新页面

从**台账列表**进入：筛选条件选好 → 导出下拉菜单新增"送检工作台" → 跳转

### 3.2 工作台布局

```
┌─────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────┤
│  [器具列表区域]                    [证书匹配区域]          │
│  ┌─────────────────────┐    ┌─────────────────────────┐ │
│  │☑ 压变 出厂编号 位置   │    │ 📁 拖拽上传证书PDF      │ │
│  │☑ 压变 7037589  B平台 │    │                         │ │
│  │☑ 温变 4190286  B平台 │    │ 匹配结果:               │ │
│  │☑ 压力表 L230822 B平台│    │ ┌─────────────────────┐ │ │
│  │☑ 压力表 180309  B平台│    │ │ ✓ 7037589 → 证书匹配 │ │ │
│  │                       │    │ │ ⚠ L230822 → 未找到  │ │ │
│  │ [全选] [删除选中]     │    │ │ ✗ 180309 → 编号差异 │ │ │
│  └─────────────────────┘    │ └─────────────────────┘ │ │

│                              └─────────────────────────┘ │

│                              ┌────────▼────────────────┐ │
│                              │  并排核对详情（选中某行） │ │

│                              └─────────────────────────┘ │

```

### 3.3 三个区域详解

#### A 区：器具列表（左侧，宽度 45%）

从预警页带入的器具列表，展示关键字段：

| 列 | 来源 | 说明 |
|----|------|------|
| ☑ | — | 勾选，默认全选 |
| 类别 | `category` |  |
| 出厂编号 | `serial_number` | **核心匹配键** |
| 安装位置 | `installation_location` | 核对用 |
| 型号 | `model` | 辅助核对 |
| 当前证书编号 | `certificate_number` | 送检前的旧证书 |
| 当前有效日期 | `valid_until` | 即将过期 |
| 匹配状态 | 实时计算 | ✓已匹配 / ⚠未匹配 / ✗编号差异 |
| 新证书编号 | 匹配结果 | 上传证书后自动填入 |
| 新检验日期 | 匹配结果 | 从证书提取 |
| 新有效日期 | 计算得出 | 根据规则计算 |

**交互**：
- 默认全选，可取消勾选（该器具不参与本次更新）
- 点击行 → 在 C 区展示并排核对详情
- 匹配状态实时更新（上传证书后自动刷新）

#### B 区：证书上传 & 匹配概览（右侧上方，宽度 55%）

```
┌──────────────────────────────────────┐
│ 📁 上传检定证书                       │
│ ┌──────────────────────────────────┐ │
│ │  拖拽PDF到此处 或 点击选择文件     │ │
│ │  支持批量上传，最多200个           │ │
│ └──────────────────────────────────┘ │
│                                       │
│ 匹配概览:  总计 30 | ✓匹配 28 | ⚠未匹配 1 | ✗差异 1 │
│                                       │
│ ┌─ 匹配详情表格 ──────────────────┐  │
│ │ 证书文件 | 证书编号 | 出厂编号 | 结果│  │
│ │ xxx.pdf | WB-...  | 4190286 | ✓  │  │
│ │ yyy.pdf | YB-...  | 7037589 | ✓  │  │
│ │ zzz.pdf | YLB-... | L23082X | ✗  │  │ ← 编号差一个字符
│ └────────────────────────────────┘  │
│                                       │
│ [全部确认并更新台账] [仅更新匹配成功的] │
└──────────────────────────────────────┘
```

**上传后自动匹配逻辑**（复用现有 batch-upload 的 `extractInspectionDate` + `findBySerialNoAndCategory`）：

1. 解析每个 PDF 文件名 → 提取证书编号、出厂编号、类别、检验日期
2. 在 A 区器具列表中按出厂编号查找匹配
3. 匹配成功 → 填入新证书编号、新检验日期、新有效日期，状态标记 ✓
4. 匹配失败 → 标记 ⚠（未找到）或 ✗（编号相似但不同）

#### C 区：并排核对详情（右侧下方，选中某行时展开）

**这是解决核心痛点的关键**——用户点击 A 区或 B 区的某一行后，展开详情对比：

```
┌─ 台账（系统记录）────────────┐  ┌─ 证书（上传文件）────────────┐
│                              │  │                              │
│ 器具类别: 压力变送器          │  │ 证书类别: 压力变送器           │
│ 出厂编号: 7037589            │  │ 出厂编号: 7037589             │
│ 安装位置: B生产二层平台       │  │ → 位置不在证书中              │
│ 型号: EJA430A                │  │ → 型号不在证书中              │
│ 旧证书号: YB-...-20250712106 │  │ 新证书号: YB-...-2026062901  │
│ 旧检验日期: 2025-07-12       │  │ 新检验日期: 2026-06-29        │
│ 旧有效日期: 2026-07-11       │  │ 新有效日期: 2027-06-28        │
│                              │  │                              │
│           [确认匹配] [跳过] [手动修改编号]  │
└──────────────────────────────┘  └──────────────────────────────┘
```

**差异高亮规则**：

| 字段 | 对比逻辑 | 高亮 |
|------|---------|------|
| 出厂编号 | 系统 vs 证书提取 | 不一致时红色高亮，显示"修正编号"按钮 |
| 检验日期 | 系统 vs 证书提取 | 证书更新 → 绿色标记"将更新" |
| 有效日期 | 系统 vs 计算预期 | 证书更新 → 绿色标记"将更新" |
| 证书编号 | 系统 vs 证书文件名 | 证书更新 → 绿色标记"将更新" |

**关键交互**：
- "确认匹配" — 标记该行为已核对，状态变 ✓
- "跳过" — 不更新该器具
- "手动修改编号" — 弹出小输入框，修改证书提取的出厂编号后重新匹配（覆盖了设计文档 `design-cert-unmatched-handling.md` 中的 force-match 场景，但在工作台上下文中更自然）

## 4. 按钮操作流

### 4.1 导出检定申请表

点击后调用现有 `GET /api/instruments/export/warning-apply`，传入当前 A 区勾选的器具 ID 列表。用户拿到申请表后送检。

### 4.2 全部确认并更新台账

点击后：
1. 遍历 A 区所有 ✓ 状态的器具
2. 对每条调用 `Instrument.updateWithHistory(id, { certificate_number, inspection_date, valid_until, certificate_file }, { source: 'inspection_batch' })`
3. 更新完成后 A 区刷新状态
4. 提示"已更新 N 条记录"

### 4.3 导出管理一览表

更新完成后，调用现有 `GET /api/instruments/export/management-summary`，导出本次更新的器具的最新一览表。

## 5. 技术实现

### 5.1 新增页面路由

```
client/src/views/InspectionWorkspace.vue    # 送检工作台页面
client/src/router/index.js                   # 新增路由 /inspection-workspace
```

### 5.2 数据传递方式

从预警页/台账页传递器具 ID 列表到工作台：

```js
// 方式：URL query 参数
router.push({ 
  path: '/inspection-workspace', 
  query: { ids: selectedIds.join(',') } 
})

// 工作台 onMounted 时根据 ids 加载完整器具数据
```

### 5.3 后端 API

**无需新增 API**。工作台使用的全部是已有接口：

| 操作 | API | 说明 |
|------|-----|------|
| 加载器具列表 | `GET /api/instruments?ids=1,2,3` | 已有（findByIds） |
| 上传证书 | `POST /api/certificate/batch-upload` | 已有，返回匹配结果 |
| 更新单条器具 | `PUT /api/instruments/:id` | 已有 |
| 导出申请表 | `GET /api/instruments/export/warning-apply?ids=...` | 已有 |
| 导出一览表 | `GET /api/instruments/export/management-summary?ids=...` | 已有 |

**可选优化**：新增 `POST /api/instruments/batch-update` 一次性更新多条，减少请求数。但如果器具数在 50 以内，逐条调用也可接受。

### 5.4 前端核心逻辑

```js
// 工作台核心状态
const instruments = ref([])          // 从 API 加载的器具列表
const certResults = ref([])          // 上传证书后的匹配结果
const selectedId = ref(null)         // 当前选中查看详情的器具 ID

// 匹配状态映射：instrumentId → certResult
const matchMap = computed(() => {
  const map = {}
  for (const cert of certResults.value) {
    if (cert.status === 'matched' && cert.matchedInstrument) {
      map[cert.matchedInstrument.id] = cert
    }
  }
  return map
})

// 每个器具的匹配状态
function getMatchStatus(instrument) {
  const cert = matchMap.value[instrument.id]
  if (!cert) {
    // 检查是否有相似但未匹配的证书
    const fuzzy = certResults.value.find(c => 
      c.status === 'unmatched' && 
      levenshteinSimilarity(c.serialNumber, instrument.serial_number) > 0.8
    )
    if (fuzzy) return 'fuzzy' // 编号差异
    return 'unmatched'
  }
  return 'matched'
}
```

### 5.5 前端依赖

- 复用现有 API 函数：`getInstruments`, `batchUploadCertificates`, `updateInstrument`
- 复用现有组件：无需新建组件，工作台本身就是一个页面
- 新增工具函数：`levenshteinSimilarity`（从 certificate.js 后端移植到前端，用于实时相似度计算）

## 7. 与现有功能的关系

| 现有功能 | 关系 |
|---------|------|
| 批量上传证书对话框 | **替代**：工作台的上传 + 匹配区域取代现有的简单对话框 |
| 证书未匹配处理 | **整合**：手动修改编号、模糊搜索匹配等功能直接在工作台 C 区完成 |
| 导入冲突检测 | **互补**：台账导入时的冲突处理和工作台的送检匹配是不同场景 |
| 证书日期校验工具 | **互补**：校验工具是事后检查，工作台是事前预防 |
| 管理一览表导出 | **保留**：在工作台按钮中直接调用 |

## 8. 对抗性审查

| # | 严重度 | 问题 | 处理 |
|---|--------|------|------|
| 1 | 中 | 工作台数据通过 URL query 传递，器具 ID 过多时 URL 过长 | 超过 30 个 ID 时改用 sessionStorage 传递，URL 只传一个批次 key |
| 2 | 中 | 用户在 A 区修改了器具信息（如修正了编号），但未点"全部确认"就关闭了页面 | 关闭前检测未保存更改，弹出确认提示 |
| 3 | 低 | 工作台同时打开两个标签页，操作同一批器具可能导致冲突 | 更新时使用 `updateWithHistory`，后操作的会基于最新数据计算 diff；不阻塞但记录 |
| 4 | 低 | 证书上传后匹配结果依赖 multer 临时文件，文件被清理后无法查看 PDF | 参考证书 PDF 在新标签页打开查看，依赖 `uploadsAuth` 鉴权 |
